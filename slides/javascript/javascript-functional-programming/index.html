What is Functional Programming?

Definition

Functional programming is a programming paradigm in which we try to bind everything in pure mathematical functions style

Big picture

  • It is a declarative type of programming style
  • Its main focus is on “what to solve” in contrast to an imperative style where the main focus is “how to solve”

Origin

  • Lambda calculus which was a formal system in mathematical logic for expressing computation
    square_sum(x, y) = x2 + y2
  • Alan Turing used it into its "state machine" to decode the Enigma system

Principle: idempotence

  • A function called with same parameters should return same value
    • Same logic as mathematical function f(x)
    • An example of non-idempotence function: Math.random()

Principle: immutability

  • Never modify the input parameters (to avoid side effects). If you want to modify one parameter:
    • Clone the parameter
    • Modify the cloned parameter
    • Return the cloned parameter

Principle: referential transparency

Never modify the variables / context outside of your function

This eliminates any chances of side effects because any variable can be replaced with its actual value at any point of execution

Pure function

A pure function should respect the three previous principles

  • Idempotence
  • Immutability
  • Referential transparency

Functions are First-Class

"First-Class" means that something is just a value

  • First-class functions are treated as first-class variable
  • The first class variables can be passed to functions as parameter
  • The first class variables can be returned from functions or stored in data structures

Functions can be Higher-Order

Higher order functions are the functions that take other functions as arguments and they can also return functions

Gains

  • Composition
  • Extensibility
  • Reusage
  • Readability
  • Testability

To have a humanized and readable code

Cons

  • Writing pure functions can be bit intimidating
  • Immutable values and recursion can lead to decrease in performance.

What kind of functions?

Pure function

Already seen

Impure function

Reversed pure function

Partial function

A partial function is a function that is not defined for all possible input values

Example

            const half = x != x / 2;

Possible input

  • number
  • string
  • void
  • object
  • array
  • symbol

Possible output

  • number
  • NaN (which is a number)
  • Uncaught TypeError

It could be not a big deal

It could be into the contract / documentation to deal only with a specified type

Total function

A total function is a function that is defined for all kind of inputs

Basically, we guarantee to return always the same output, despite the specified inputs

Example in Java World:

            public Optional<User> findUser(string uuid) {
   // We will return an optional, not directly a user
   // So we could have an empty optional (with isPresent())
   // Or we could have an optional with the user
}

Functional Patterns

Basics needs - bind

Bind

Goal: return a function with a specified context

Why: prevent context overriding, expecially when we set as function parameter our function

Nota bene: there is now a bind function on functions

Implementation

            function bind(funcToBind, context) {
   return function () {
      return funcToBind.apply(context, arguments);
   };
}

Usage

            this.a = { name: 'John' };
const greet = (age) => `Hi ${this.name}, you have ${age} years old.`
const boundGreet = bind(greet, { name: 'Jack' });

boundGreet(42); // Hi Jack, you have 42 years old.

Basics needs - partial

Partial

Goal: return a function to harldy wrapped some parameters

Why: function composition

Implementation

            function partial(funcToPartial, args...) {
   return function () {
      return funcToPartial.apply(this, args.concat(Array.from(arguments)));
   };
}

Usage

            const greet = (name, age) => `Hi ${name}, you have ${age} years old.`
const boundGreet = partial(greet, 'Jack');

boundGreet(42); // Hi Jack, you have 42 years old.

Data needs - map

Map

Goal: convert a data to a new format

Why: adapt the data to your need

Implementation

            function map(arr, callback) {
   const mapping = [];
   const index = 0;
   arr.forEach((entry) => {
      mapping.push(callback.apply(this, [ entry, ++index ]);
   };
   return mapping;
}

Usage

            const data = [1, 2, 3];

map(data, (val) => {
   return { num: val };
}); // [{num:1},{num:2},{num:3}]

Data needs - flatten

Flatten

Goal: convert an array of arrays (and more) to a one-level array

Why: ease to iterate

Example of functional recursion

Implementation

            function flatten(arr) {
   return arr.reduce(
    (flattenArray, entry) => {
       return flattenArray.concat(Array.isArray(entry) ? flatten(entry) : entry);
    },
    []
   );
}

Usage

            const array = [1, [2, [3, [4]], 5]];

flatten(array); // [1, 2, 3, 4, 5]

Advanced needs - debounce

Debounce

Goal: delay the execution of a function

Why: avoid to execute many time the function into a short delay

Implementation

            function debounce(funcToDebounce, delay = 150) {
   let token = null;
   return function () {
      const currentArgs = arguments;
      if (token) {
         clearTimeout(token);
      }

      token = setTimeout(() => {
            token = null;
            funcToDebounce.apply(this, currentArgs);
      }, delay);
   };
}

Usage

            const greet = (name, age) => console.log(`Hi ${name}, you have ${age} years old.`)
const boundGreet = debounce(greet, 500);

boundGreet('Jack', 42);
setTimeout(() => boundGreet('William', 18), 250);
boundGreet('John', 24);
// One log: Hi William, you have 18 years old

Advanced needs - compose

Compose (or pipelining)

Goal: compose a set of functions, to chain them and return a new function

Why: increase composition

Implementation

            function compose(funcs...) {
   return function () {
      let lastArgs = arguments;
      let lastReturn = undefined;

      funcs.forEach((func) => {
         lastReturn = func.apply(this, lastArgs);
         lastArgs = [ lastReturn ];
      });

      return lastReturn;
   };
}

Usage

            const data = [2, 3, null, 1, 42, false];
const myPipeline = compose(compact, initial, rest, sort);

myPipeline(data); // [1, 3]
// similar to sort(rest(initial(compact(data))))

ES6 feature: pipeline operator

Which provides natively the "pipeline"

            const data = [2, 3, null, 1, 42, false];
data |> compact |> initial |> rest |> sort; // [1, 3]

// Or with indentation
data
   |> compact
   |> initial
   |> rest
   |> sort; // [1, 3]

God pattern - chaining

Chaining (A.K.A transduction)

Goal: transform boilerplate code

Why: increase composition, readiblity and performance

Example

I have a list of users. I want

  • To filter major users
  • To filter users with lastname starting with 'b'
  • To get the third user

Boilerplate implementation

Hard lisibility

            let expectedUser = null;
let potentialUser = null;
let time = 0;

for(const user : users) {
   if (user.age > 18 && user.lowercase().startWith('b')) {
      potentialUser = user;
      index++;
   }

   if (index == 3) {
      expectedUser = potentialUser;
      break;
   }
}

ES6 implementation

Gain lisibility. But if I have 1000 users, I could iterate 2001 times

            const filtered = users
   .filter((user) = user.age >= 18)
   .filter((user) = user.lastname.lowercase().startWith('b'));

const expectedUser = filtered.length > 2 ? filtered[2] : null;

Functional implementation

Gain lisibility, and performance

            const expectedUser = chain(users)
   .filter((user) = user.age >= 18)
   .filter((user) = user.lastname.lowercase().startWith('b'))
   .get(3)
   .value()

Extrem functional implementation

            const expectedUser = chain(users)
   .filter(isUserMajor)
   .filter(partialRight(isUserLastNameStartWith, 'b'))
   .get(3)
   .value()

What about strongly typing?

Typing is a nice to have

Typing allows writing code

  • based on propositions
  • based on proof

In deep: proposition

            declare function add(x: number, y: number): number;

Proposition: If x and y are numbers, a number exists

In deep: proof

            function add(x: number, y: number): number {
   return x + y;
}

Proof: x + y is a proof that a number exists

Types are propositions

Programs are proofs

Based on "Curry-Howard Correspondence"

Used in Haskell programs

https://wiki.haskell.org/Curry-Howard-Lambek_correspondence

So, what does it mean?

Let the type system suggest the implementation

If I typed

            function head<T>(list: T[]): T {
   return list;
}

I will have

            f[ts] Type 'T[]' is not assignable to type 'T'. (parameter) list: T[]

With typing, this is much easier to have "total function"

With typing we could always return a value

With functional meaning

With less defensive code

At the end, we could have functors or monads

Example

            async function fetchUser(username: string): Promise<Either<FetchError, User!>> {
   const res = await fetch(`https:!//api.sugarpirate.com/users/${username}`);
   if (!res.ok) { return left(new FetchError(`[${res.status}] ${res.statusText}`)) }
   return right(await res.json());
}
